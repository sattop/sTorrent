# sTorent: поэтапная работа AI-агента

## Цель

Этот документ нужен, чтобы ChatGPT, Codex или другой AI-агент разрабатывал `sTorent` поэтапно, а не пытался сразу сделать весь проект. Каждый этап должен иметь понятный результат, проверку и короткий отчет.

## Главные правила для AI

1. Сначала читать документы из `docs/`.
2. Не перепрыгивать через этапы.
3. Перед кодом уточнять архитектуру и выбранный стек.
4. После каждого этапа запускать доступные проверки.
5. Не удалять чужие изменения без явного разрешения.
6. Не добавлять функции подмены трафика под чужие сервисы.
7. Не коммитить секреты, токены, пользовательские `.torrent` файлы, логи и скачанные данные.
8. Каждый этап завершать понятным списком сделанного и списком следующих шагов.

## Рекомендуемый порядок работы

### Этап 0: подготовка

Задачи:

- прочитать все документы в `docs/`;
- проверить структуру проекта;
- выбрать стек: Tauri или Electron;
- создать базовую структуру репозитория;
- подготовить `README.md`, `.gitignore`, `LICENSE`;
- подготовить базовую i18n-структуру по `docs/10-localization-i18n.md`;
- настроить GitHub-репозиторий, если пользователь дал доступ.

Результат:

- проект можно открыть локально;
- есть понятная структура папок;
- нет лишних сгенерированных файлов.

Проверка:

- проект запускается или хотя бы собирает пустой шаблон;
- `.gitignore` исключает `node_modules`, `dist`, пользовательские данные и секреты.

### Этап 1: прототип интерфейса

Задачи:

- главный экран загрузок;
- моковые торренты;
- темная и светлая темы;
- диалог добавления `.torrent` и magnet;
- экран настроек;
- переключение языка интерфейса;
- UI профилей загрузки;
- UI Speed Doctor на моковых данных;
- базовая навигация.

Результат:

- пользователь видит будущий интерфейс клиента;
- можно оценить дизайн без реального torrent-core.

Проверка:

- UI запускается локально;
- нет визуальных перекрытий;
- главный экран понятен на обычном desktop-разрешении;
- язык переключается и сохраняется;
- профили можно выбрать в UI;
- Speed Doctor показывает моковый диагноз без реального torrent-core;
- если есть WebUI, проверить адаптивность на мобильной ширине.

### Этап 2: базовый torrent-core

Задачи:

- подключить выбранную torrent-библиотеку;
- добавить `.torrent`;
- добавить magnet;
- получить metadata;
- начать загрузку;
- поставить на паузу;
- продолжить загрузку;
- удалить torrent из списка;
- проверить хеши.
- добавить ручные профили загрузки.

Результат:

- клиент умеет реально скачивать тестовую легальную раздачу.

Проверка:

- тестовая загрузка стартует;
- прогресс обновляется;
- пауза и продолжение работают;
- после перезапуска состояние сохраняется.
- профиль применяется к новой загрузке.

### Этап 3: файлы, категории и теги

Задачи:

- выбор файлов внутри раздачи;
- выбор папки сохранения;
- категории с путями;
- теги;
- фильтры по статусу, категории и тегам;
- безопасное удаление torrent/data.

Результат:

- пользователь контролирует, что именно скачивается и куда сохраняется.

Проверка:

- выбранные файлы скачиваются корректно;
- категории не ломают путь сохранения;
- удаление данных требует подтверждения.

### Этап 4: сеть и приватность

Задачи:

- DHT/PEX/LSD настройки;
- private torrent режим;
- входящий порт;
- UPnP/NAT-PMP;
- лимиты скорости;
- выбор сетевого интерфейса;
- proxy settings;
- BitTorrent protocol encryption;
- диагностика сети.
- Speed Doctor v1: лимиты, очередь, сиды, трекер, порт, место на диске.

Результат:

- пользователь может настроить стабильное сетевое поведение без небезопасной имперсонации трафика.

Проверка:

- лимиты применяются;
- private-раздачи уважают private flag;
- диагностика показывает понятные результаты;
- Speed Doctor объясняет причины и предлагает действия;
- выбранный интерфейс корректно ограничивает соединения, если эта функция поддерживается core.

### Этап 5: автоматизация

Задачи:

- watch folders;
- избранные папки;
- правила сидирования;
- RSS-ленты;
- правила автозагрузки;
- расписание лимитов;
- hooks/webhooks после MVP.

Результат:

- пользователь может автоматизировать повторяющиеся действия.

Проверка:

- `.torrent` из watch folder добавляется автоматически;
- правило RSS не скачивает дубли;
- правила сидирования не удаляют данные без подтверждения.

### Этап 6: WebUI и remote API

Задачи:

- локальный WebUI;
- пароль;
- responsive-интерфейс;
- API для основных действий;
- ограничения доступа по IP;
- документация API.

Результат:

- клиентом можно управлять с другого устройства в локальной сети.

Проверка:

- WebUI выключен по умолчанию;
- без пароля доступ запрещен;
- мобильная ширина выглядит нормально;
- пауза/продолжение через WebUI работает.

### Этап 7: Windows `.exe`

Задачи:

- настроить сборку `sTorent Setup.exe`;
- добавить иконку;
- заполнить metadata приложения;
- проверить установку и удаление;
- подготовить portable-версию после MVP.

Результат:

- пользователь может скачать и запустить `.exe`.

Проверка:

- `.exe` запускается двойным кликом;
- установщик создает ярлык;
- удаление работает через Windows;
- настройки пользователя не удаляются без подтверждения.

### Этап 8: GitHub и релизы

Задачи:

- настроить GitHub Actions;
- проверять сборку на push;
- собирать `.exe` на tag `v*`;
- публиковать GitHub Release;
- прикреплять `SHA256SUMS.txt`;
- готовить changelog.

Результат:

- релиз создается автоматически после публикации git-тега.

Проверка:

- workflow проходит без ошибок;
- `.exe` прикреплен к релизу;
- checksum совпадает;
- в артефакты не попали секреты и пользовательские данные.

## Формат отчета после каждого этапа

AI должен отвечать так:

```text
Этап: <номер и название>

Сделано:
- ...

Проверено:
- ...

Файлы:
- ...

Следующий этап:
- ...
```

Если этап не завершен, AI должен явно написать:

```text
Этап не завершен.
Причина:
- ...

Что нужно сделать дальше:
- ...
```

## Правило остановки

AI должен остановиться и спросить пользователя, если:

- нужно выбрать между Tauri и Electron, а в проекте нет явного решения;
- нужен токен GitHub или доступ к репозиторию;
- нужно подписывать Windows `.exe` платным сертификатом;
- действие может удалить данные пользователя;
- тесты показывают ошибку, которую нельзя безопасно исправить без изменения архитектуры.

## Минимальный промпт для запуска разработки

```text
Прочитай документы в папке docs и разрабатывай sTorent строго поэтапно по docs/09-ai-step-by-step-workflow.md. После каждого этапа запускай проверки, кратко отчитывайся и не переходи к следующему этапу, если текущий не работает. Не реализуй маскировку трафика под чужие сервисы. Windows-релиз должен собираться как sTorent Setup.exe и публиковаться через GitHub Actions.
```
